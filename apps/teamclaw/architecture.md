## 1. 产品定位

TeamClaw 是一个**桌面客户端**（Windows + macOS），定位为多 Agent 协作的项目交付平台。用户通过微信或桌面端与系统交互，由 Main Agent 主管调度多个专业 Agent（PM、Designer、Architect、Coder、DBA、DevOps）协同完成软件开发任务。

### 核心理念
- **不造轮子**：Agent 编排能力内嵌 OpenClaw，复用其成熟框架
- **人机协作**：关键节点需用户审核确认，Agent 不自行决定重大事项
- **微信优先**：通过 OpenClaw Bot 桥接微信，文字交互为主

## 2. 技术选型

| 层级 | 技术方案 | 说明 |
|------|---------|------|
| 桌面框架 | **Tauri 2.0** | 轻量、安全、跨平台，Rust 后端 |
| 前端 UI | **React + TypeScript** | Tauri webview 渲染 |
| 本地数据库 | **SQLite** (via Drizzle ORM) | 零配置，数据存本地 |
| Agent 引擎 | **OpenClaw** | 内嵌调用，负责 Agent 编排与执行 |
| 微信桥接 | **OpenClaw Bot** → 微信适配层 | 文字交互，审核二次确认 |
| 向量存储 | **ChromaDB**（本地） | 知识库 RAG 检索 |
| 实时通信 | **本地事件总线** | Agent 状态推送、任务进度更新 |
| 包管理 | **pnpm** | Monorepo 工作区管理 |

## 4. Agent 角色定义

| Agent | 职责 | 输入 | 输出 |
|-------|------|------|------|
| **Main** | 主管，接收任务、拆解、调度、汇总提问 | 用户需求 | 任务分配、进度汇报 |
| **PM** | 需求分析、PRD 编写 | 需求描述 | PRD 文档 |
| **Designer** | UI/UX 设计 | PRD | HTML/CSS 前端代码 |
| **Architect** | 技术架构、API 设计 | PRD | 架构文档、API Schema |
| **CoderA** | 前端开发 | 设计稿 + 架构 | 前端代码 |
| **CoderB** | 后端开发 | 架构文档 | 后端代码 |
| **DBA** | 数据库设计、迁移 | 架构文档 | Schema、Migration |
| **DevOps** | 部署、CI/CD、环境配置 | 架构文档 | 配置文件、部署脚本 |

### Agent 编排模式 — DAG
- 任务以 **DAG（有向无环图）** 方式编排
- 可并行的 Agent 同时执行，有依赖关系的串行执行
- 用户可在关键节点干预：**暂停、修改、跳过、回滚、中止、插入**

## 5. 数据模型（核心实体）

```
Project
  ├── id, name, description, status, created_at
  │
  ├── Conversation (1:N)
  │     ├── id, project_id, title, status, created_at
  │     │   status: creating | clarifying | splitting |
  │     │           executing | reviewing | iterating |
  │     │           delivering | archived
  │     │
  │     └── Message (1:N)
  │           ├── id, conversation_id, role, content, agent, timestamp
  │           │   role: user | agent | system
  │           │
  │           └── Attachment (1:N)
  │                 ├── id, message_id, type, path
  │
  ├── Task (1:N)
  │     ├── id, project_id, parent_id, title, status
  │     │   status: pending | assigned | in_progress |
  │     │           blocked | reviewing | done | failed
  │     │   assignee: agent_role
  │     │   depends_on: [task_id]
  │     │
  │     └── Artifact (1:N)
  │           ├── id, task_id, type, path, version
  │
  └── Knowledge (1:N)
        ├── id, project_id, source, embedding, metadata
```

## 7. 微信交互规范

### 消息流程
```
用户(微信) → OpenClaw Bot → TeamClaw Main Agent → 分发给专业 Agent
                                                          ↓
用户(微信) ← OpenClaw Bot ← Main Agent(汇总) ← Agent 结果收集
```

### 交互规则
- **仅文字交互**，不做语音消息
- **审核二次确认**：所有需要用户确认的操作，微信端需回复「确认」才能执行
- **Agent 并发提问**：多个 Agent 的问题由 Main Agent 汇总后统一提问，避免轰炸用户
- **消息格式**：结构化卡片 + 关键信息摘要

## 8. 安全与约束

- 本地数据优先，敏感信息不上云
- Agent 执行代码在沙箱环境中运行
- 所有代码修改需用户审核后才写入项目
- 微信端操作有操作频率限制
- 对话上下文有长度上限，超出部分归档到知识库
